## 1. Per-instance LB & gap decomposition

### Per-pair min-tof matrix (Probe E)

Built `min_tof_cheap[i,j]` = minimum tof such that `compute_transfer(i, j, t_start, tof) ≤ 100 m/s`, scanning t_start ∈ [0, 192d] × tof ∈ [0.05, 8.0]d (20×80 grid, parallel, ~3 min wall).

Key stats:
- **Cheap-edge graph density: 5.9 %** (138 / 2352 directed pairs reach dv ≤ 100)
- **Median out-degree: 2** (41/49 nodes have ≤4 cheap neighbors)
- **No node BFS-reaches all 48 others via cheap edges** (max reach 40 of 49)

### Component structure (the structural backbone)

Undirected cheap-edge graph has **4 connected components**:
- comp 0: 40 nodes (the "main body")
- comp 1: {4, 11, 17}
- comp 2: {16, 27, 32}
- comp 3: {18, 23, 34}

**Therefore the Hamiltonian path requires K − 1 = 3 exception bridges to connect components.** Budget allows 5 → 2 extras for in-component optimization.

The bank actually uses 5 exception bridges. **Structural minimum is 3.** This is a notable observation: bank may be over-using exceptions inside components (legs 24, 27, 40, 45 within comp 0; only leg 2 is the inter-comp bridge 18→46).

### Quantitative gap decomposition

| Cause | Estimate | Reasoning |
|---|---|---|
| C1: in-component hamiltonian-path slack | ~20 d minimum (LB sum of (size−1) smallest cheap edges in each comp) | structural |
| C2: per-leg tof slack on current bank perm | 44.48 d (bank sum 129.30 − per-leg-min sum 84.82) | time-coupling forces dep timing |
| C3: idle waits between legs | 13.62 d | walk_perm_chrono `wait_dt=1.0` |
| C4: K−1 inter-component exception bridges | ~0.5 d each × 3 = 1.5 d | bank inter-comp bridge tof = 0.337 d for leg 2 |
| C5: 2 extra exceptions used inside components | unknown — could shorten or worsen | bank chose to use; need cross-validation |
| **UNEXPLAINED** | — | per-instance global LB requires LP/MILP relaxation |

**Per-leg LB on bank perm = 84.82 d.** Vs R3 = 111.76 d. So **even if every per-leg tof shrank to its global minimum, bank makespan would be 84.82 d + idle = ~98 d** — UNDER R3 by ~14 d.

The bank's 142.92 d is **57.4 d above its own per-leg LB**, not 31 d above R3. The R3 makespan of 111.76 d is achievable with this perm IF the time-coupling slack is closed.

But Probe D (SLSQP joint polish on bank perm) showed `(times, tofs)` are at a **local optimum**: SLSQP can't shave a single day. The per-leg-min targets are mutually inconsistent because each pair's optimal t_start differs.

**Conclusion**: gap is structural in (perm, schedule) JOINTLY. Need either: (a) different perm where per-leg-mins are temporally compatible, (b) global TSP-LP / time-expanded MILP that respects coupling, or (c) heavy LNS that escapes the local optimum.

## 3. Silent-reject path audit (Bug-Surfacing P6)

Reviewed `src/esa_spoc_26/ch2_findtransfer_greedy.py` and `ch2_insert_lns.py`:

### Reject path R1 — `find_earliest_transfer` returns (None, None)

Triggered when no tof ∈ [max(min_tof, 0.05), tof_window] yields dv ≤ threshold. **No instrumentation**: count + dv-distribution of failed attempts not logged. Calls per greedy step = O(n × n_steps) = thousands; rejects are silent.

### Reject path R2 — `walk_perm_chrono` hard-fails after wait loop

If neither cheap nor exception nor (cheap-or-exc after wait_steps × wait_dt) finds a feasible transfer, returns `ok=False` with leg index `k`. Caller (insert_lns) discards the entire chain. **No logging of why** — could be tof_window too small, dv_thr too tight at this t_start, or chained idle pushed t past `max_time`.

### Reject path R3 — `compute_transfer` returns +inf

When Lambert fails (both cw directions) at this (t_start, tof). Used both as "no transfer" and "infinite cost". **Cannot distinguish from genuinely high-cost transfer** without an outer exception handler.

**Recommendation R1+R2+R3**: instrument with a per-call counter dict
`{('R1_cheap',i,j): n_reject, ('R1_exc',i,j): n_reject, 'R2_walk_fail': [...], 'R3_lambert_fail': n}` and dump after each run. We're searching huge spaces with no visibility into what's being thrown away.

## 4. Hostile-default audit (Bug-Surfacing P4)

For each default in the construction pipeline, ask: "what if this were maximally adversarial?"

### D1: `tof_window=18.0`, `n_steps=180` in `walk_perm_chrono`

Grid step = 18.0 / 180 = **0.1 d** (~144 min). Vs spec `dt_min = 0.001 d` (~86 s). The grid is **100× coarser than the minimum tof allowed by spec**. For tight near-coplanar transfers, the spec allows tof < 0.1 d, but our grid never tries < 0.05 d (the hardcoded floor).

**Adversarial scenario**: a pair where the optimal tof is 0.07 d. Our grid jumps from 0.05 (rejected) to 0.15 (accepted) — losing 0.08 d of slack per leg. Across 48 legs, worst-case is 48 × 0.05 d = **2.4 d** of forced overshoot.

### D2: `wait_dt=1.0` in `walk_perm_chrono` (DEFAULT — banged the bank)

When greedy is stuck, advance time by **1.0 d**. The bank's 13.62 d of idle = ~14 wait calls × 1.0 d. If wait_dt were 0.1 d, idle could be 1.4 d (potentially). Per Probe C (re-walk with finer wait): naive parameter sweep regressed instead of improving, indicating the function is **non-monotone in `wait_dt`** — likely because waits commit to suboptimal earliest-feasible at intermediate t.

**This is the largest single hostile default.** It accounts for the entire 13.62 d idle in the bank.

### D3: `wait_steps=12` in `walk_perm_chrono`

Caps wait time at `12 × 1.0 = 12 d`. If a transfer needs longer waiting (e.g., destination tomato in an unfavorable phase), function gives up. Adversarial: any pair requiring a 13+ d wait silently fails.

### D4: `max_revs=20` in `KTTSP.__init__`

UDP spec allows up to 20 revolutions. We use the spec default. Adversarial: if optimal pair uses 5+ revs, lambert returns 41 solutions per cw, increasing compute and potentially numerical issues. **Verify** by re-running with `max_revs=5` and comparing.

### D5: greedy starting node (`start=0` in greedy, `start=34` in bank construction)

Bank's 142.92 d found from `start=34`. **Only one starting node tried in current pipeline.** All 49 candidate starts produce different greedy trajectories. **Confirmed in pipeline `ch2_findtransfer_greedy.search`** that all 49 starts ARE tried — but the LNS / chain insertion only used start=34's partial. Other starts may have been mis-ranked by partial-coverage heuristic.

### D6: cluster boundaries in `cluster_missing_nodes`

The chain-insertion pipeline (`insert_lns`) handles missing nodes as a contiguous chain. **Hostile**: if cheap-edge-connected sub-graph among the missing has multiple components, contiguous insertion is suboptimal.

## 5. Diversity-of-method cross-validation (Bug-Surfacing P3)

The bank was produced by:  greedy_findxfer(start=34) → insert_lns chain insertion.

**Have we cross-validated this with an independent solver?** Looking at `src/esa_spoc_26/ch2_*.py`, we have **18+ Ch2 solvers** (CP-SAT, OR-tools Hamilton, MILP, BiteOpt, fcmaes, SADE, ALNS, ILS, sub-tour-insert, hierarchical, ...). The bank record indicates many converged to 142.92 d — but **were any of them ever run with anything other than the BANK-DEFAULT find_earliest_transfer parameters?** Per code inspection: no. They all share `walk_perm_chrono(wait_dt=1.0)` somewhere in the pipeline.

**The "many methods converge to 142.92" is therefore evidence of common-substrate failure, not architecture ceiling.** This is exactly the trap that the bug-surfacing methodology warns against (aggregate plausibility).

Per anti-oscillation discipline, before proposing levers, name the structural alternatives and which decomposition row they address.

| Lever | Row addressed | Predicted magnitude | Failure mode if wrong |

| L1: instrument silent reject paths in walk_perm_chrono, find_earliest_transfer | Investigation enabler (Bug-Surfacing P6) | n/a, surfaces evidence | low — pure observability |

The natural rank-ordering by expected ROI:
1. **L1 (instrument)** — required infrastructure for everything else
2. **L4 (time-expanded MILP/CP-SAT)** — only way to know the true LB; resolves whether 31-d gap is reachable
3. **L3 (component-aware insertion)** — addresses structural exception-waste; concrete + cheap
4. **L2 (wait_dt sweep + cross-multi-start)** — small expected gain but mandated by per-instance evidence
5. **L5 (heavy LNS)** — only if L1-L4 don't close the gap
6. **L6 — skip** (Probe D ruled out)
